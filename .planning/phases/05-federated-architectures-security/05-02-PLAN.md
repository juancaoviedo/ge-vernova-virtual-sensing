Mental Model — Edge Security Beyond "Just TLS"`** — 2-3 sentence framing: an autonomous K3s substation fleet needs three guarantees beyond an encrypted channel — that the code it runs is genuine (OTA signing), that the hardware booted untampered (TPM attestation), and that each workload is who it claims to be (SPIFFE/SPIRE). TLS encrypts the pipe; these answer "should this node be trusted at all?"

3. **`## 2. OTA Update Integrity (Awareness)`** — use the EXACT FED-03 template from PATTERNS:
   - `**What it is:**` signed firmware/software image; build pipeline signs with a private key in an HSM; the edge device verifies the signature before applying; only validly-signed images install.
   - `**The concrete grid threat it counters:**` a malicious actor pushes a firmware image to a fleet of substation K3s nodes that disables protection relays or exfiltrates PMU data; without signing, any image on the update channel is applied; with signing, the attacker must compromise the signing key, not just the channel.
   - `**Interview sentence:** *"OTA signing means an attacker must compromise the signing key, not just the update channel — the edge node rejects any image that doesn't pass signature verification before execution."*`
   - `**→ Bridge to your work:**` Juan has shipped OTA updates (CV); the upgrade is to add a signing step in the pipeline and a signature-check at the node — same mechanism, grid-safety guarantee.

4. **`## 3. TPM Attestation (Awareness)`** — same template:
   - `**What it is:**` a TPM 2.0 hardware security chip; during boot each stage measures the next into Platform Configuration Registers (PCRs) — a tamper-resistant running hash; remote attestation lets a verifier (fog aggregator / SPIRE server) challenge the device for PCR values signed by its endorsement key; matching known-good PCRs admits the node to the FL round.
   - `**The concrete grid threat it counters:**` a physically tampered or OS-compromised relay/K3s node requests to join the federated aggregation; without attestation the aggregator can't tell a clean node from a rooted one and accepts its gradients; with attestation the node must prove its boot-chain hash matches known-good before any update is accepted.
   - `**Interview sentence:** *"TPM attestation proves a relay is running unmodified trusted code before it joins the control loop — a hardware root-of-trust, not a password."*`
   - `**→ Bridge to your work:**` Juan runs K8s/edge fleets; the upgrade is TPM-attested node admission so only verified-clean nodes join the aggregation.
   (Awareness only — do NOT explain PCR indices, the quote structure, or attestation protocol internals.)

5. **`## 4. SPIFFE / SPIRE Workload Identity (Awareness)`** — same template, plus the Pitfall-5 correction:
   - `**What it is:**` SPIFFE = CNCF-graduated standard for cryptographic workload identity; each process (K3s pod, NATS subscriber, FastAPI service) gets a short-lived X.509 SVID whose SAN URI encodes a SPIFFE ID (`spiffe://trust-domain/path`); SPIRE is the implementation — a SPIRE server issues SVIDs, SPIRE agents attest each node (optionally via TPM) before issuing credentials; SVIDs auto-rotate, no static shared secrets.
   - CRITICAL (Pitfall 5): explicitly state SPIFFE is **workload identity, not just TLS** — TLS encrypts the channel; SPIFFE identifies *which specific workload* is on each end and rotates that identity automatically. Include this as a `**Not just TLS:**` line.
   - `**The concrete grid threat it counters:**` an attacker who compromised a device on the NATS network (or stole a long-lived API key) tries to impersonate a real substation's EKF service and inject false state estimates into the aggregation; without workload identity the aggregator can't distinguish the real substation-7 EKF pod from a spoofed one; with SPIFFE/SPIRE the legitimate pod holds a short-lived SVID the attacker can't forge, and the spoofed connection is rejected at the mTLS handshake.
   - `**Interview sentence:** *"SPIFFE gives each workload a short-lived, auto-rotating cryptographic identity so a spoofed node can't impersonate a real IED — identity, not just an encrypted channel."*`
   - `**→ Bridge to your work:**` Juan runs an MQTT device fleet with K8s identity; the upgrade path is SPIFFE/SPIRE workload identity replacing static keys.
