## Summary

Five source files reviewed: the new `fault_sim.py` driver, new additions to `influx.py`
(`write_fault_step`), new constants in `config.py`, the `pyproject.toml` script entry,
and the Grafana dashboard JSON. Focus was correctness bugs in pandapower mutation, NaN
handling, InfluxDB schema type consistency, and determinism — not web or auth security.

The overall structure is sound. Block-transition topology mutations are correct, the
dead-zone traversal assertion is mathematically correct for the IEEE 33-bus topology,
the OLTC pin (D-12) is properly scoped to the ISO block with `run_control=False`, the
D-17 read-back query covers all 40 steps with correct range arithmetic, and the Grafana
time ranges are verified to cover every step timestamp. The schema is fully additive
(writes only to `FAULT_EVENT_BUCKET`, never touching `STATE_BUCKET`).

One BLOCKER was found: a crash-on-print in the D-16 gate. Four warnings covering a
dead-sgen double-write, a missing `energised` tag on sgen points, an undeclared
`networkx` direct dependency, and silent exception swallowing in tie selection.

## Critical Issues

### CR-01: D-16 Else-Branch Crashes With `ValueError` When `iso_loads` Is Empty

**File:** `system1-measurement-source/src/ieee33/fault_sim.py:530-533`

**Issue:** The D-16 served-load gate checks `if iso_loads and min(iso_loads) >= pre_served_load`.
When `iso_loads` is empty, the condition is `False` (correct — no issue to record), and
execution falls to the `else` branch, which runs this print:

```python
print(
    f"D-16 served-load drop OK: iso_min={min(iso_loads) if iso_loads else 'N/A':.4f} MW "
    f"< pre_fault={pre_served_load:.4f} MW"
)
```

The inline conditional `min(iso_loads) if iso_loads else 'N/A'` evaluates to the string
`'N/A'`, and then `:.4f` is applied to that string, raising:

```
ValueError: Unknown format code 'f' for object of type 'str'
```

In practice `FAULT_ISO_STEPS = 7` means `iso_loads` will always have 7 entries and this
path is unreachable in the current configuration. However, should `FAULT_ISO_STEPS` ever
be set to 0 (e.g., for a diagnostic run or a config change), the script exits with an
unhandled `ValueError` at the success-reporting stage, masking the actual result.

**Fix:** Split the format string so `'N/A'` is never fed to `:.4f`:

```python
iso_min_str = f"{min(iso_loads):.4f}" if iso_loads else "N/A"
print(
    f"D-16 served-load drop OK: iso_min={iso_min_str} MW "
    f"< pre_fault={pre_served_load:.4f} MW"
)
```

## Warnings

### WR-01: Dead-Zone Sgen Written Twice Per ISO Step (Double-Write in `write_fault_step`)

**File:** `system1-measurement-source/src/ieee33/influx.py:469-486`

**Issue:** During the `faulted_isolated` block, `sgen 0` (bus 17, inside the dead zone)
is set `in_service=False` before `runpp`. Pandapower sets `res_sgen.at[0, 'p_mw'] = 0`
and `res_sgen.at[0, 'q_mvar'] = 0` for out-of-service sgens (it does NOT emit NaN —
this is confirmed pandapower 3.x behaviour). The function then:

1. **Energised-sgen loop** (line 469): iterates `res_sgen`, which includes row 0 with
   `p_mw=0`, and writes `Point("sgen", sgen_id="0", p_mw=0.0, q_mvar=0.0)`.
2. **Dead-zone zero-fill loop** (line 479): iterates `dead_sgen_ids=[0]` and writes
   `Point("sgen", sgen_id="0", p_mw=0.0, q_mvar=0.0)` again.

Both points land in the same batch with identical measurement+tag+timestamp. InfluxDB
last-write-wins within a batch, and since both writes carry the same values the final
stored data is correct. But the energised-sgen loop's write for sgen 0 is conceptually
wrong: it implies sgen 0 is a live, energised generator during ISO, when it is not.

**Fix:** Skip dead sgens in the energised-sgen loop:

```python
dead_sgen_set = set(dead_sgen_ids)
for idx, row in res_sgen.iterrows():
    if idx in dead_sgen_set:
        continue          # dead-zone sgen handled by zero-fill loop below
    points.append(
        Point("sgen")
        .tag("sgen_id", str(idx))
        .field("p_mw",   float(row["p_mw"]))
        .field("q_mvar", float(row["q_mvar"]))
        .time(timestamp)
    )
```

### WR-02: `sgen` Measurement Lacks `energised` Tag — Schema Asymmetry Breaks System 2 Filtering

**File:** `system1-measurement-source/src/ieee33/influx.py:468-486`

**Issue:** The `bus` and `line` measurements both carry an `energised` tag (`"1"` for
live, `"0"` for dead-zone zero-fill). The `sgen` measurement has no such tag in either
the energised loop or the dead-zone zero-fill loop. This means:

- System 2 queries on `sgen` cannot distinguish sgen 0 with `p_mw=0` because the PV
  array is inactive (zero solar irradiance) from sgen 0 with `p_mw=0` because the bus
  is dead-isolated.
- The schema docstring in `write_fault_step` describes bus and line with the `energised`
  extension but is silent on sgen — the omission is unintentional given the stated
  forward-contract purpose of the function.

**Fix:** Add the `energised` tag to the sgen writes, mirroring bus and line:

```python
# Energised sgens
for idx, row in res_sgen.iterrows():
    if idx in dead_sgen_set:
        continue
    points.append(
        Point("sgen")
        .tag("sgen_id", str(idx))
        .tag("energised", "1")           # <-- add
        .field("p_mw",   float(row["p_mw"]))
        .field("q_mvar", float(row["q_mvar"]))
        .time(timestamp)
    )

# Dead-zone sgen zero-fill
for dead_sgen_idx in dead_sgen_ids:
    points.append(
        Point("sgen")
        .tag("sgen_id", str(dead_sgen_idx))
        .tag("energised", "0")           # <-- add
        .field("p_mw",   0.0)
        .field("q_mvar", 0.0)
        .time(timestamp)
    )
```

Note: adding a new tag to an existing InfluxDB series requires migration or a clean
bucket if this bucket has been written before, since InfluxDB v2 enforces consistent
tag sets per series. Given this is a development setup with idempotent re-run
semantics, a `delete --bucket fault_event` before re-running `fault-sim` is sufficient.

### WR-03: `networkx` Used Directly but Not Declared as a Direct Dependency

**File:** `system1-measurement-source/pyproject.toml:5`

**Issue:** `fault_sim.py` imports `networkx` directly (`import networkx as nx`,
`from pandapower.topology import ...` uses it internally too). `networkx` is not listed
in `[project.dependencies]`. It is currently installed transitively because pandapower
declares it in its own `install_requires`. A future pandapower release that inlines the
graph logic or switches graph libraries would silently break the direct `import networkx`
call.

**Fix:** Add `networkx` as an explicit direct dependency:

```toml
dependencies = [
  "pandapower==3.4.0",
  "networkx>=3.0",          # <-- add; used directly in fault_sim.py
  "influxdb-client==1.50.0",
  ...
]
```

### WR-04: `_select_restore_tie` Swallows All Exceptions Without Logging

**File:** `system1-measurement-source/src/ieee33/fault_sim.py:159`

**Issue:** The tie-selection loop catches all exceptions with a bare `except Exception: continue`.
If a tie fails because of a pandapower crash (wrong parameter type, missing column,
etc.) rather than non-convergence, the exception text is silently discarded. When all
three candidate ties fail and `RuntimeError` is raised at line 162, the error message
gives no indication of which exception occurred or on which tie.

**Fix:** Log the exception before continuing:

```python
except Exception as exc:
    print(f"  tie_idx={tie_idx}: exception during probe ({type(exc).__name__}: {exc})")
    continue
```
