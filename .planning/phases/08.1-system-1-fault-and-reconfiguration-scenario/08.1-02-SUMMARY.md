# Phase 08.1 Plan 02: Fault Simulator Runner Summary

**One-liner:** Deterministic 40-step fault-and-reconfiguration runner (fault_sim.py) — frozen evening-peak operating point, graph-traversal dead-zone isolation, tap-pinned NR during ISO block, tie-34 restoration with nx.is_tree radiality gate, 40 snapshots written to fault_event with full D-05 continuity.

## What Was Built

One new file created (`system1-measurement-source/src/ieee33/fault_sim.py`, 598 lines, purely additive):

### Task 1 — Scaffold: dead-zone traversal + run_fault_step + evening-peak anchor (commit f852330)

- `_compute_dead_buses(net)`: graph traversal via `ptop.create_nxgraph` + `nx.node_connected_component` from the slack bus; returns sorted dead-bus indices and asserts == `sorted(config.FAULT_DEAD_BUS_IDX)` as a model-integrity guard (D-07).
- `run_fault_step(net, trafo_idx, base_p, base_q, load_pu, solar_pu, wind_pu, phase, peak_tap)`: frozen operating-point per-step runner. Mirrors `sim.run_step` line-for-line for load/DG scaling; D-12 OLTC pin — during `faulted_isolated` pins `net.trafo.at[trafo_idx,"tap_pos"] = int(peak_tap)` and runs `pp.runpp(..., run_control=False)` because `OLTC_REF_BUS=17` is inside the dead zone. Returns `state["res_bus"]` filtered to in-service buses only (see deviation).
- Module-level constants: `DEAD_LINE_IDS = [7,8,9,...,16]`, `DEAD_SGEN_IDS = [0]`, `DEAD_SHUNT_IDS = [0]` (sgen 0 and shunt 0 are both on bus 17).
- `main()` scaffold: `build_enhanced_33bus()` once, `influx.get_client()` + `wait_for_influx()` + `ensure_bucket(client, config.FAULT_EVENT_BUCKET)`, `influx.read_profiles(client)` sorted + `.iloc[config.FAULT_EVENING_PEAK_STEP]` to extract anchor timestamp and frozen `load_pu/solar_pu/wind_pu`.

### Task 2 — 40-step walk, per-step write, gates, console table (commit 83e70b2)

- `_select_restore_tie(load_pu, solar_pu, wind_pu, base_p, base_q)`: iterates `[config.FAULT_TIE_IDX] + [other TIE_LINE_IDX]`; each candidate builds a fresh net (via `build_enhanced_33bus()`) to avoid mutating the main net, applies restored topology, runs `pp.runpp(..., run_control=True)`, checks converged + `nx.is_tree` + `energised_vm.min() >= 0.94`. Returns `(tie_idx, vmin)`. On the locked evening-peak operating point, tie 34 wins with vmin = 0.9861 pu.
- **Phase plan:** `phases = [FAULT_PHASE_PRE]*13 + [FAULT_PHASE_ISO]*7 + [FAULT_PHASE_RST]*20`; `timestamps = [anchor_ts + timedelta(seconds=3*i) for i in range(40)]`.
- **Evening-peak tap capture:** First pre_fault step run with `run_control=True`; `peak_tap = state["tap_pos"]` = -2 (BOOST, confirmed matches planner expectation).
- **Block transitions (mutate in-place at boundary):**
  - ISO: line 7 + DEAD_LINE_IDS[1:] + FAULT_DEAD_BUS_IDX + DEAD_SGEN_IDS + DEAD_SHUNT_IDS → `in_service=False`; dead-bus set computed by `_compute_dead_buses` and asserted.
  - RST: DEAD_LINE_IDS[1:] + FAULT_DEAD_BUS_IDX + DEAD_SGEN_IDS + DEAD_SHUNT_IDS → `in_service=True`; faulted line 7 stays out; chosen tie → `in_service=True`.
- **Per-step write-call id sets:**
  - pre_fault: `dead_bus_ids=[], dead_line_ids=[], dead_sgen_ids=[], tie_is_closed=False`
  - faulted_isolated: `dead_bus_ids=[8..17], dead_line_ids=[7,8..16], dead_sgen_ids=[0], tie_is_closed=False`
  - restored: `dead_bus_ids=[], dead_line_ids=[7], dead_sgen_ids=[], tie_is_closed=True` — D-05: faulted line 7 zero-filled in EVERY restored step for series continuity; in-zone lines 8-16 re-energised normally.
- **D-14 gate:** `net.converged` check + `state["res_bus"]["vm_pu"].isna().any()` check per step.
- **D-15 radiality gate:** For each restored step — asserts tie_in_service=True AND fault_in_service=False AND `nx.is_tree(ptop.create_nxgraph(net))`.
- **D-16 gates:** Post-loop served-load drop (`iso_min < pre_served_load`) + restored-block vmin band (>=0.95 clean, >=0.94 flag, <0.94 hard issue).
- **D-17 read-back:** Flux query fault_event for bus_id="0" vm_pu over anchor window; asserts exactly 40 points.
- **D-18 console table:** Header + 40 rows of `step/phase/vmin/load/tap/dead` (width-padded).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] pandapower includes dead buses with NaN vm_pu in res_bus**

- **Found during:** Task 2 execution — first `uv run fault-sim` returned NaN for faulted_isolated steps
- **Issue:** The PATTERNS.md comment stated "when `in_service=False`, pandapower excludes those buses from `res_bus`" — empirically false. Pandapower 3.x sets dead-bus rows in `res_bus` to NaN vm_pu rather than excluding them. This caused the D-14 NaN gate to fire on every ISO step even though the power flow converged correctly.
- **Fix:** In `run_fault_step()`, added `bus_in_service_mask = net.bus["in_service"]` and `res_bus_energised = net.res_bus[bus_in_service_mask].copy()` before returning. The `state["res_bus"]` dict key now contains only energised buses. Updated NaN check in `main()` to use `state["res_bus"]["vm_pu"].isna()` rather than `net.res_bus["vm_pu"].isna()`. Also used `state["res_bus"]` for `energised_vm` in the D-15/D-16 tracking.
- **Files modified:** `system1-measurement-source/src/ieee33/fault_sim.py`
- **Commit:** 83e70b2

**2. [Rule 1 - Bug] InfluxDB query_data_frame returns tables in tag-alphabetical order, not time order**

- **Found during:** Task 2 verification — plan's read-back script `list(df2['phase'])[:13]` returned `['faulted_isolated'] * 7` first because the `phase` tag value `faulted_isolated` sorts alphabetically before `pre_fault`
- **Issue:** The plan's verification script does not call `.sort_values("_time")` on the dataframe before extracting the `phase` list. `query_data_frame` concatenates InfluxDB result tables (split by series = tag combination) in the order the server returns them — alphabetically by tag value — not in time order.
- **Fix:** The underlying data is correct (confirmed by adding `.sort_values("_time")` in an alternate verification run). Documented as a known limitation of `query_data_frame`; the production gates in `fault_sim.py` itself use the D-17 bus_id="0" vm_pu read-back which is a single-series query (not multi-table) and returns 40 points correctly. The plan's verification script passed when run with the corrected sort.
- **Impact:** The plan's `<automated>` verify script will fail on the `phases[:13]` assertion if run verbatim — not because the data is wrong but because pandas concatenates multi-table results by table order. All substantive data requirements are satisfied.
