# Phase 8.1: System 1 Fault and Reconfiguration Scenario — Specification

**Created:** 2026-06-24
**Ambiguity score:** 0.17 (gate: ≤ 0.20)
**Requirements:** 10 locked

## Goal

Produce a **separate, re-runnable, quasi-steady-state fault-and-reconfiguration dataset** on the
existing System 1 IEEE 33-bus model — replicating the **Section-8 fault → isolate → tie-restore**
sequence from Meteab/Tousi/Omran (2025) — as a **40-step (2-min @ 3-s), three-block** sequence
(`pre-fault → faulted-isolated → reconfigured-restored`) at a frozen evening-peak operating point,
with the **complete power-flow state plus topology/event metadata captured per snapshot** and
persisted to a **dedicated InfluxDB bucket and its own provisioned Grafana dashboard**, to
stress-test the future System 2 state estimator.

## Background

Phase 8 shipped **System 1** in `system1-measurement-source/`: an enhanced IEEE 33-bus radial
network (`build_enhanced_33bus` in `network.py`), a 96-step (15-min) quasi-static day driver
(`sim.py`), InfluxDB helpers (`influx.py`, buckets `profiles`/`state`), validation (`validate.py`),
Docker Compose (InfluxDB 2.9.1 + Grafana 11.6.15), and a provisioned Grafana dashboard — all via
`uv`. The model has high-penetration DER (~6.8 MW, DG at buses 18/22/25/33), a ±5% line-drop OLTC,
per-line ACSR ampacity, and **three normally-open tie-lines** (pandapower line idx 32 `bus20↔7`,
33 `bus8↔14`, 34 `bus11↔21` = article 21↔8, 9↔15, 12↔22).

There is **no fault/contingency capability today** — System 1 only walks a healthy daily profile and
never changes topology beyond the OLTC tap. The source paper
(`.planning/research/articles/self_healing_1.pdf`) simulates a 3-phase fault in Section 8 (the
article bus 8→9 line), isolates it (open Sw-8/Sw-9), and **closes a tie-switch to restore the
de-energised downstream sections 9–17**, with DGs supporting the restored zone — analysed in CYME
as a sequence of steady-state load flows (and separately, an EMT transient in Simulink which is
*not* relevant here). This phase replicates that **steady-state reconfiguration sequence** on our
System 1 model. The matching reconfiguration in our model: fault the `art 8→9` line (pandapower
line idx 7) → de-energise the art 9–18 main-feeder zone → close tie idx 34 (`art 12↔22`) to
back-feed it from the healthy lateral.

This is the first of a multi-system design Juan described (context only — not this phase):
**System 1** (behaviour simulator, this + Phase 8) → **Measurement System** (sampling) →
**System 2** (state estimator) → **System 3** (self-healing loop). This phase builds **only** the
System 1 fault scenario.

## Requirements

1. **Separate fault/reconfiguration runner**: A new entry point produces the fault scenario by
   reusing System 1, without touching the 96-step day.
   - Current: only `sim.py` (healthy 96-step day) exists; no fault/contingency runner
   - Target: a new runner (e.g. `uv run fault-sim`) reuses `build_enhanced_33bus()` and the
     `influx.py` helpers to produce the fault-and-reconfiguration sequence; `sim.py`, the `state`
     bucket, and the existing dashboard are left unchanged
   - Acceptance: the new entry point runs from one command and completes; `git`/`grep` shows
     `sim.py` and the existing `state`-bucket/dashboard artifacts are unmodified by this phase

2. **Frozen evening-peak operating point**: The whole window holds load/DER fixed at the
   heaviest-load operating point.
   - Current: no fault scenario; the day driver varies load/DER every step
   - Target: load and DER are fixed at the evening-peak step's values for all 40 steps; the only
     things that change across the window are the fault and the switch actions
   - Acceptance: across the 40 snapshots, the per-bus load and per-DG nameplate-scaled injection are
     constant (identical at every in-service step); only topology (line/tie in-service) changes

3. **Section-8 fault + isolation (out-of-service modelling)**: A 3-phase permanent fault on the
   article 8→9 line is isolated, de-energising the downstream main-feeder zone.
   - Current: no fault modelling; all non-tie lines always in service
   - Target: the faulted line (pandapower idx 7, article 8→9) is set out-of-service to represent
     isolation; the resulting unsupplied downstream zone (article buses 9–18 / pandapower idx 8–17,
     plus any sub-laterals left sourceless) is **set out-of-service (de-energised — no voltage/flow
     rows)** during the faulted-isolated block
   - Acceptance: in the faulted-isolated snapshots, `pp.runpp` converges on the *healthy* remainder,
     line idx 7 is out-of-service, and the de-energised buses are out-of-service (absent from
     `res_bus`); the served-load total drops by the isolated zone's load

4. **Tie-line reconfiguration restores the de-energised zone**: Closing a normally-open tie
   re-feeds the isolated zone from a healthy path while keeping the network radial.
   - Current: tie-lines are always open (radial); no reconfiguration logic
   - Target: in the restored block, a normally-open tie (reference: idx 34, `art 12↔22`; plan-phase
     may select a different feasible tie) is closed to back-feed the previously de-energised zone;
     the faulted line stays out-of-service; the network remains radial (no loop created)
   - Acceptance: in the restored snapshots the tie is in-service, the faulted line is out-of-service,
     all previously-dead buses are back in service with valid voltages, `pp.runpp` converges, and a
     radiality check passes (no closed loop)

5. **40-step, three-block, 3-second timeline**: The window is an ordered sequence of quasi-static
   snapshots spanning before, during, and after the event.
   - Current: no event timeline exists
   - Target: 40 snapshots at 3-second spacing (~2 min) in three ordered blocks —
     pre-fault (healthy) → faulted-isolated (zone dead) → reconfigured-restored — with a sustained
     isolated window of ≥1 snapshot before restoration; exact block sizes are a plan detail
     (reference split ≈ 13 / 7 / 20)
   - Acceptance: the dataset contains exactly 40 timestamped snapshots at 3-s spacing; each carries a
     phase tag in {`pre_fault`, `faulted_isolated`, `restored`}; all three tags are present and the
     order is pre_fault → faulted_isolated → restored

6. **Full state + topology/event metadata per snapshot**: Each snapshot captures System 1's full
   power-flow state plus the topology and event context needed by an estimator.
   - Current: System 1's `state` bucket stores per-entity power-flow state but no topology/event
     context (topology is static there)
   - Target: each snapshot stores the System-1 full state (per-bus vm_pu/va_degree, per in-service
     line P/Q/loading/losses, slack P/Q, DG P/Q, OLTC tap/shift, system scalars) **plus** per-snapshot
     topology/event metadata: which lines/ties are in-service, which buses are energised vs dead, and
     the phase tag
   - Acceptance: for a chosen snapshot, a query returns the full per-entity state for energised
     entities **and** the topology metadata (faulted line id, tie closed y/n, list/flag of
     de-energised buses, phase tag)

7. **Dedicated InfluxDB bucket**: The scenario writes to its own bucket, separate from System 1's.
   - Current: only `profiles` and `state` buckets exist
   - Target: a new bucket (e.g. `fault_event`) holds the 40-snapshot fault dataset; `profiles` and
     `state` are untouched; the bucket is created programmatically (existence-checked)
   - Acceptance: after a run, the new bucket contains the 40-snapshot series; a query on `state`
     still returns the original 96-step day unchanged

8. **Provisioned Grafana dashboard for the scenario**: A separate dashboard renders the event.
   - Current: one provisioned dashboard (the 96-step day) exists
   - Target: a second auto-provisioned Grafana dashboard (over the new bucket) renders the fault
     scenario — at minimum bus-voltage envelope, OLTC tap, the served-load / slack feed-in drop and
     recovery, and a visual marker of the three phases; the original dashboard is unchanged
   - Acceptance: Grafana lists two dashboards; the new one renders the 40-step event (voltage
     collapse of the isolated zone → recovery on restoration) with no manual setup

9. **Restoration heals voltages into band, deterministically**: The restored state is a valid,
   in-band (or documented near-band) operating point, and runs are reproducible.
   - Current: no restoration result exists
   - Target: in the restored block, all energised buses return to **0.95–1.05 pu (target)**; if the
     single-tie back-feed cannot hold 0.95 — as in the article's own 0.94 pu result — the run records
     the actual minimum and the criterion is met at **≥ 0.94 pu with the shortfall flagged** (not a
     hard failure); re-running yields identical snapshots
   - Acceptance: restored-block min voltage ≥ 0.95 (or ≥ 0.94 with an explicit logged flag); two
     consecutive runs produce identical snapshots (overwrite-in-place, no duplicates)

10. **Runbook + qualitative article comparison**: The scenario is documented and qualitatively
    related to the paper.
    - Current: the README covers only the 96-step day
    - Target: the README documents how to run the fault scenario and open its dashboard; a short note
      states what is replicated from the article (Sect-8 fault → isolate → tie-restore) and that the
      comparison is **qualitative** — restoration succeeds and losses/voltages move in the directions
      the paper reports — with **no numerical match expected** (our DG placement 18/22/25/33 + ±5%
      OLTC differ from the paper's PSO DGs at 6 & 32)
    - Acceptance: README contains the fault-scenario run steps and the qualitative-comparison note;
      following it from a clean state reproduces the 40-snapshot dataset and renders the dashboard

## Boundaries

**In scope:**
- A new fault/reconfiguration runner that reuses System 1 (`network.py`, `influx.py`, Docker stack, uv)
- 40-step, 3-block, 3-s quasi-static sequence frozen at evening peak
- Section-8 fault modelled as a faulted line out-of-service + de-energised zone out-of-service
- Tie-line reconfiguration that restores the de-energised zone (radiality preserved)
- Full state + topology/event metadata + phase tag captured per snapshot
- A dedicated InfluxDB bucket and a separate provisioned Grafana dashboard
- Determinism + README runbook + a qualitative article-comparison note

**Out of scope:**
- The Measurement System (sampling System 1 into estimator inputs) — a later phase
- System 2 (state estimator) and System 3 (self-healing loop / automatic switch decisions) — later phases
- EMT / transient / sub-cycle dynamic simulation (the paper's Simulink waveform) — pandapower is a
  steady-state tool and the estimator runs at quasi-static cadence; not needed
- Protection-relay / breaker timing and the literal 0.02 s isolation / 60 ms restoration transient —
  we model the ordered topology *sequence*, not the sub-cycle timing
- Numerical match to the article's loss/voltage tables, and any PSO DG re-placement (DGs at 6 & 32)
  to make an article-faithful variant — different model on purpose
- Multiple or sequential faults — a single permanent fault only
- Any change to System 1's 96-step day, its `state`/`profiles` buckets, or its existing dashboard

## Constraints

- Reuse System 1: pandapower `build_enhanced_33bus()`, `influx.py` helpers, the Docker Compose
  InfluxDB 2.9.1 + Grafana 11.6.15 stack, and `uv`. Python 3.12.
- Quasi-steady-state only: every snapshot is a converged Newton-Raphson power flow; the de-energised
  zone is represented with `in_service=False` (pandapower will not converge on a sourceless island
  with live load, so isolated elements MUST be taken out of service).
- Operating point frozen at the evening-peak step (load/DER held constant across the window).
- New bucket distinct from `profiles`/`state`; new dashboard distinct from the 96-step one.
- Deterministic: identical inputs → identical 40 snapshots; no wall-clock randomness.
- The OLTC line-drop controller and the in-zone DG (DG at bus idx 17 sits inside the isolated zone)
  may be leveraged to support the restored-zone voltage.

## Acceptance Criteria

- [ ] One command runs the fault scenario end-to-end and writes 40 timestamped snapshots at 3-s spacing
- [ ] `sim.py`, the `state` bucket, and the original Grafana dashboard are unchanged by this phase
- [ ] Every snapshot carries a phase tag in {`pre_fault`, `faulted_isolated`, `restored`}; all three present, in order
- [ ] Load/DER are constant across all in-service snapshots (only topology changes)
- [ ] In the faulted-isolated block: faulted line (idx 7) out-of-service, de-energised zone out-of-service, power flow converges on the healthy remainder, served load drops
- [ ] In the restored block: a tie is closed, faulted line stays out-of-service, all previously-dead buses are re-energised, power flow converges, radiality holds
- [ ] Restored-block min voltage ≥ 0.95 pu (or ≥ 0.94 pu with an explicit logged shortfall flag)
- [ ] Each snapshot stores full per-entity state **and** topology/event metadata (faulted line, tie state, de-energised buses, phase tag)
- [ ] The new InfluxDB bucket holds the 40-snapshot series; a query on `state` still returns the original 96-step day
- [ ] Grafana shows a second auto-provisioned dashboard rendering the event (isolation collapse → restoration recovery) with no manual setup
- [ ] Re-running produces identical snapshots (deterministic)
- [ ] README documents the run + the qualitative-comparison note (no numerical match expected vs the article)
