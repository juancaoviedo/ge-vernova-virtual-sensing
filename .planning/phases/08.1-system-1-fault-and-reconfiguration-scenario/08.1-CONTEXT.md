# Phase 8.1: System 1 Fault and Reconfiguration Scenario - Context

**Gathered:** 2026-06-24
**Status:** Ready for planning

<domain>
## Phase Boundary

A **separate, re-runnable, quasi-steady-state fault-and-reconfiguration dataset** on the
existing System 1 IEEE 33-bus model — replicating the article's **Section-8 fault → isolate →
tie-restore** sequence as a **40-step (2-min @ 3-s), three-block** quasi-static sequence
(`pre_fault → faulted_isolated → restored`) at a frozen evening-peak operating point. Each
snapshot captures System 1's full power-flow state plus topology/event metadata, persisted to a
**dedicated InfluxDB bucket** with its **own provisioned Grafana dashboard**, to stress-test the
future System 2 estimator.

Built entirely by reusing System 1 (`build_enhanced_33bus()`, `influx.py` helpers, the Docker
stack, `uv`). **No change** to `sim.py`, the `state`/`profiles` buckets, or the existing dashboard.

</domain>

<decisions>
## Implementation Decisions

### Metadata schema (forward contract for System 2)
- **D-01:** Topology/event metadata lives in a **new `event` measurement** (not on `system`). This
  keeps System 1's `bus`/`line`/`sgen`/`system` measurement schema **byte-identical** (clean
  forward contract) and gives System 2 one clearly-named place for event context.
- **D-02:** The phase label is stored as an **indexed Influx tag** `phase` ∈ {`pre_fault`,
  `faulted_isolated`, `restored`} so Grafana can group/color/annotate by phase and System 2 can
  filter cheaply. Put the `phase` tag on the `event` point (and may also tag other points to ease
  Grafana region markers — planner's call).
- **D-03:** The `event` measurement carries, per snapshot: `faulted_line_id` (=7), `tie_closed`
  (bool/int), `tie_id` (the closed tie, else none), `n_dead_buses` (int field), and `dead_buses`
  (comma-joined id string, e.g. `"8,9,10,..."`). Count drives quick Grafana stats; the explicit
  list lets System 2 reconstruct exact topology.

### Dead-bus / dead-line representation (continuous series)
- **D-04:** **Simulation layer:** the dead zone is modelled with `in_service=False` so `pp.runpp`
  converges on the healthy remainder (dead entities are absent from `res_bus`/`res_line` — this
  satisfies SPEC req 3's power-flow acceptance).
- **D-05:** **Writer layer:** for de-energised entities, the writer **zero-fills explicit rows**
  into InfluxDB rather than leaving gaps — `bus` rows get `vm_pu=0`, `va_degree=0` (and the bus's
  load registered as 0); `line` rows (faulted line + in-zone lines) get `p/q/loading/losses=0`.
  This produces a **continuous before → 0 → after series** per entity so the full evolution is
  visible and directly comparable to System 2's reconstruction. *(Refinement of SPEC reqs 3 & 6:
  "absent from res_bus" refers to the pandapower result object; the persisted dataset gets explicit
  zeros.)*
- **D-06:** Each `bus` point (and each `line` point) carries an explicit **`energised` flag (1/0)**
  so consumers unambiguously distinguish a de-energised zero-fill from a live reading (0 pu is
  already physically impossible, but the flag removes any inference). Applies to dead lines too.

### Fault / tie pair
- **D-07:** Fault is **hard-locked at pandapower line idx 7 (article 8→9)** — the exact Section-8
  fault the paper simulates. The isolated dead zone is the downstream main-feeder zone (article
  9–18 / pandapower idx 8–17, plus any sub-laterals left sourceless — exact set computed from
  topology by the planner/runner).
- **D-08:** Restoration tie is **idx 34 (article 12↔22) as primary**; plan-phase verifies it
  restores the zone radially to ≥0.95 pu (or ≥0.94 flagged) and **may fall back to another feasible
  tie only if idx 34 cannot** restore. Balances article-fidelity with the voltage outcome.

### Timeline & timestamps
- **D-09:** Three-block split = **13 pre-fault / 7 faulted-isolated / 20 restored = 40** snapshots
  (the SPEC reference split): healthy baseline, sustained outage window (≥1 required), long restored
  tail to show recovery settle.
- **D-10:** The 40×3-s series is anchored at the **evening-peak step's real OPSD UTC timestamp**
  (the operating point we freeze on), then +3 s per step. Conceptually consistent (it *is* the
  evening-peak snapshot) and fully deterministic — no wall-clock randomness.

### In-zone DG & OLTC behavior
- **D-11:** The **in-zone DG (sgen at bus 17)** goes **out-of-service with the dead zone** during
  the faulted-isolated block (zero-filled like its bus), then **reconnects and contributes** when
  the tie restores the zone. (A sourceless out-of-service island can't host a grid-following
  inverter in steady-state pandapower — modelling it as still injecting would not converge.)
- **D-12:** **OLTC landmine:** `OLTC_REF_BUS = 17` is **inside** the dead zone, so the line-drop
  controller cannot regulate it while isolated. Resolution: **pin the tap at the evening-peak
  (pre-fault) position during the faulted-isolated block**, then **resume line-drop control after
  restoration** so it actively boosts the back-fed zone toward ≥0.95. Simple, deterministic, avoids
  the dead-reference crash. *(Reuses sim.py's `forced_tap`/`run_control=False` mechanism.)*

### Dashboard panels (beyond SPEC minimums)
- **D-13:** SPEC minimums required: bus-voltage envelope, OLTC tap, served-load/slack feed-in drop
  & recovery, and a 3-phase visual marker. **Add all four extras:**
  (1) **dead-zone bus voltage time-series** (per-bus lines for art 9–18 showing collapse to 0 →
  recovery — the headline before→0→after visual);
  (2) **dead-bus count panel** (n_dead_buses from `event`, 0 → isolated count → 0);
  (3) **restored min-voltage stat** with threshold color (green ≥0.95, amber ≥0.94, red below);
  (4) **topology/event state table** (faulted line id, tie closed y/n, tie id, phase per snapshot).

### Validation & determinism gates (runner enforces)
- **D-14:** **Power-flow convergence per snapshot** — every `pp.runpp` converges; NaN-voltage check
  on energised buses (mirrors sim.py's per-step guard).
- **D-15:** **Radiality after restore** — post-restoration topology is radial (no closed loop), tie
  closed + faulted line open. *Kept as a hard gate* (locked by SPEC req 4, matches the article's
  FLISR restoration, cheap correctness guard that catches an accidental mesh). In this scenario it
  holds near-automatically because the faulted line is out of service.
- **D-16:** **Served-load drop + restored band** — served load drops in the faulted block; restored
  min voltage ≥0.95 pu (or ≥0.94 with an explicit logged shortfall flag — not a hard failure).
- **D-17:** **40-point readback + determinism** — after write, assert exactly 40 snapshots in the
  new bucket; two consecutive runs produce identical snapshots (overwrite-in-place, no duplicates).

### Runner CLI & output
- **D-18:** **Zero-arg, deterministic runner.** `uv run fault-sim` runs the locked scenario with no
  flags and prints a per-step console table (step, phase, vmin, served load, tap, dead count) in
  sim.py's style. Fully reproducible.

### Claude's Discretion (naming — confirmed by user)
- New InfluxDB bucket: **`fault_event`** (existence-checked via `ensure_bucket()`).
- Runner entry point: **`uv run fault-sim`** → `ieee33.fault_sim:main` (new `[project.scripts]` entry).
- Grafana dashboard JSON: **`grafana/provisioning/dashboards/ieee33-fault-event.json`**, title
  **"IEEE 33-Bus — Fault & Reconfiguration"** (auto-provisioned via existing `default.yml`).
- Exact computation of the dead-bus set (graph traversal from the isolated edge) and exact block
  boundaries within the 13/7/20 split are planner details.

<code_context>
## Existing Code Insights

### Reusable Assets
- `build_enhanced_33bus()` → returns `(net, trafo_idx)` fully assembled; the fault runner builds
  this once, then mutates `in_service` flags per block.
- `influx.ensure_bucket()` — idempotent bucket creation for the new `fault_event` bucket.
- `influx.write_state_step()` — the per-entity write path; the fault writer mirrors its
  measurement/field/tag naming for `bus`/`line`/`sgen`/`system`, adds the `energised` flag + new
  `event` measurement, and zero-fills dead entities.
- `sim.run_step(forced_tap=…, run_control=False)` — exact mechanism to pin the OLTC tap during the
  isolated block (D-12).
- Per-step console table + NaN/band checks in `sim.main()` — pattern for the fault runner's output
  and validation gates.

### Established Patterns
- **Article-N = pandapower idx N-1** (0-indexed everywhere): fault art 8→9 = line idx 7; dead zone
  art 9–18 = bus idx 8–17; tie art 12↔22 = line idx 34.
- Deterministic overwrite-in-place: identical timestamp+tag keys overwrite (no duplicates on rerun).
- Per-step writes (no buffering all snapshots) — keep this.
- Tap sign on this transformer: **negative tap_pos = BOOST** (raises voltage); positive = BUCK.

### Integration Points
- New runner module `ieee33/fault_sim.py` (+ `[project.scripts] fault-sim`), new `fault_event`
  bucket, new `grafana/.../ieee33-fault-event.json`. All additive — System 1's day driver, buckets,
  and dashboard are untouched (verifiable by `git`/`grep`).

</code_context>

<specifics>
## Specific Ideas

- User's headline goal for the zero-fill design: a **continuous per-entity series** (voltage/current
  before the fault → 0 during → restored after) so the dataset shows "the full evolution of the
  system" and can be **directly compared against System 2's reconstructed state** at every bus.
- Replicate the article's **FLISR self-healing** shape (open faulted section, close one tie, DGs
  support restored zone) — qualitatively, on our model, no numerical match expected.
